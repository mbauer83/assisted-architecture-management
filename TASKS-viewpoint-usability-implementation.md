### P-05-INV: what `traversal` does the GUI builder emit today? — ANSWERED 2026-07-17

**Answer: the GUI builder emits NO `traversal` key at all; the server parses the
default `direct`. The GUI does NOT write `derived` undisclosed.**

Evidence chain (all verified against the live backend + code at 3eea9b0):

1. `tools/gui/src/domain/viewpointCriteria.ts` — `IncidentNode` has no traversal
   field; `mkIncident()` creates direction/negate/connectionCriteria/endpointCriteria
   only. `tools/gui/src/domain/viewpointCriteriaSerialization.ts:76`
   (`incidentToMapping`) never writes `traversal`.
2. **Latent round-trip defect (new, sharper than F5):** `incidentFromMapping`
   (viewpointCriteriaSerialization.ts:151) never READS `traversal` either — loading
   any definition whose incident predicate carries `traversal: derived` into the GUI
   editor and re-saving silently strips the field, rewriting semantics derived→direct.
   P-05's selector work must fix both directions and add a regression test.
3. **The run's PH1 "derived-only match" diagnosis was WRONG in mechanism.**
   Reproduced PH1's exact ad-hoc query (`POST /api/viewpoints/execute`, no traversal
   key): 1 match = `BOB@1780547142.cyfMBB.artifact-grouping` ("Artifact Grouping").
   That entity has ZERO outgoing modeled connections; its match comes via an INCOMING
   `archimate-association` from `ART@1780547760.NKi5Hz.group-directory` (technology
   domain). `direction_matches` (`src/domain/viewpoint_criteria_evaluation.py:28`)
   deliberately passes ANY direction for symmetric connection types, and
   `archimate-association` is the ontology's only `symmetric: true` type
   (`src/ontologies/archimate_4/connections.yaml:90`). So the predicate evaluated
   DIRECT edges exactly as the default says; the undisclosed semantics are the
   symmetric-type direction normalization, not derived traversal. (The derived
   witness chain the report cited — probe-PL1-deps-store.json — belongs to the
   `element-dependencies` MODULE definition, whose `include_connected` declares
   `traversal: derived, max_hops: 4`; different mechanism, different surface.)

Consequences adopted:
- **Migration ("current behavior"):** GUI-authored saved predicates have no traversal
  key ⇒ semantics were always `direct`. The upgrade-CLI default-stamping writes
  `traversal: direct` explicitly — a pure no-semantic-change stamp. No saved recipe
  ever executed derived incident predicates via the GUI.
- **P-01 wording:** the per-predicate summary names the traversal in words
  ("has a direct connection…"); no "derived" caveat is needed for legacy GUI recipes.
- **New disclosure gap (symmetric direction) owned where summaries are owned:** the
  phrase "has an outgoing connection" is untruthful for associations by design.
  Minimal truthful-summary handling decided under P-01/P-05 (see Decisions D-1).

## Decisions

- **D-3 (P-06 alternates):** R-16 asked for "access to alternates where several
  witnesses exist" — the derivation engine (`relationship_reachability.py`) retains
  exactly ONE best witness per (source, target) pair by construction, so there are no
  alternates to expose. `witness_steps` emits the retained chain; exposing alternates
  would require changing the derivation engine's result retention, out of P-06 scope.
- **D-2 (P-03 last_updated):** dropped from the projection per the proposal's own
  either/or — verified absent from the read-index EntityRecord; indexing it with
  defined modification-semantics is a separate change if wanted.
- **D-1 (symmetric-direction disclosure, from P-05-INV):** the summary generator's
  truthful-semantics mandate (P-01 applied principle) extends to the symmetric-type
  direction normalization discovered during the investigation: when a directional
  incident predicate is rendered, the summary discloses that associations match in
  either direction — smallest wording change, no engine change, no new proposal
  scope. The engine behavior itself (symmetric normalization) is intentional and
  stays; flagged to the owner as a sharpened restatement of F5's mechanism.
