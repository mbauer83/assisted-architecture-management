## Context

Assurance work — hazard analysis, unsafe control actions, compliance obligations, risks, control structures — produces sensitive analytical content about a system's weaknesses. The architecture model, by contrast, lives in broadly-readable git repositories designed for sharing and promotion. Storing assurance findings alongside the model would expose what was analysed, and what was found, to everyone with repository access — violating the principle that [Assurance Content Is Confidential by Default](../../../projects/motivation-narrative/model/motivation/principle/PRI@1780655839.hVWaYW.assurance-content-is-confidential-by-default.md). The main alternatives considered were:

- **Model findings as motivation-domain entities in the architecture repository**: reuses existing tooling, but leaks confidential content into the shared git tier, gives findings the wrong lifecycle (promotion, broad readability), and permanently records sensitive analysis in git history.
- **Back-references from architecture entities to assurance artifacts**: convenient navigation, but every back-reference is a git-visible pointer revealing that a finding exists — the same leak through diffs and history.
- **A single universal storage interface for both stores**: one abstraction, but the architecture store needs rich query and copy-on-write semantics while the assurance store needs CRUD, unlocking, and classification — a shared interface forces lowest-common-denominator semantics on both.
- **A separate, gated, optionally-encrypted assurance store with one-way persisted references** (chosen).

## Decision

Assurance artifacts are *not* architecture-model entities. They form their own artifact family, persisted in the [Confidential Assurance Repository](../../../projects/assurance/model/business/business-object/BOB@1780656241.SgtAoz.confidential-assurance-repository.md) — a third tier distinct from engagement and enterprise — accessed through the pluggable [Confidential Assurance Store](../../../projects/assurance/model/application/application-component/APP@1780656431.E0fzqZ.confidential-assurance-store.md) port. The default backend is SQLCipher, embedded and encrypted with a key auto-managed in the OS keychain, so encryption is simply on with no ceremony; team and advanced backends plug in behind the same port. Cryptography is delegated to vetted stores, never bespoke.

The capability is gated fail-closed: the assurance MCP servers and GUI features do not exist unless a store is configured *and* unlocked. No assurance content is reachable in a normal deployment.

Cross-references are persisted one-way only: assurance records point to architecture entities by artifact id, and the architecture repository never stores back-references — even edges whose semantic direction reads architecture→assurance are stored on the assurance side. The architecture repository's git history therefore leaks nothing about what was analysed. Navigation is still bidirectional at runtime: back-references shown on architecture views are computed in the read model for unlocked sessions, never persisted.

## Consequences

**Positive:**
- Confidential analysis never enters a shared git tier; diffs, history, and promotions carry no trace of assurance activity.
- Encrypted-by-default with zero ceremony — the safe path is the easy path for small teams without dedicated security infrastructure.
- Fail-closed gating means read-only or unconfigured sessions cannot even discover the capability's content surface.
- Each store keeps interfaces fit for purpose: rich query semantics for the model, CRUD-plus-unlock-plus-classification for assurance.
- Runtime-computed back-references give analysts full bidirectional navigation without persistence-side leakage.

**Negative / risks:**
- Cross-store referential integrity is weaker: an architecture entity can be deleted while assurance records still reference it, requiring a tolerant resolver for dangling references.
- Two storage stacks mean two backup, migration, and operational stories instead of one.
- The OS-keychain key binding requires a recovery-key export safeguard; losing both means losing the store.
- At-rest encryption does not cover the AI exposure vector — content decrypted for an unlocked session still flows to an LLM via MCP; that vector is owned by classification and tool-boundary redaction, a separate mechanism that must be maintained in parallel.
