## Context

A running instance has at most two physical repository roots — one engagement repository and, optionally, one enterprise repository. The in-memory artifact index over these roots was historically keyed by whichever *combination* of roots a caller requested (`{engagement}`, `{enterprise}`, `{engagement, enterprise}`), so the backend could hold several independent, fully-populated index instances covering the same physical directory. The same content was parsed, held in memory, and SQLite-indexed up to three times over — and, worse, the instances could diverge: a write visible in one instance and stale in another produced a real consistency bug in exactly this shape. The main alternatives considered were:

- **Per-caller / per-combination index instances** (status quo): no coordination needed between callers, but it is the proven bug class — divergent stale indexes for the same root — and pays a multiplied parse and memory cost for identical content.
- **A dedicated WorkspaceTopology abstraction** to own the root pair and hand out indexes: adds a named concept, but the two-root constraint is already encoded structurally in the workspace-resolution function's return type — a new class would duplicate a constraint the codebase already makes unrepresentable.
- **N-root generalization** (index over an arbitrary list of roots): flexible, but the workspace is defined as one engagement plus one enterprise repository; generality nothing needs would blur the invariant that makes the design checkable.
- **One canonical index per resolved physical root, plus a stateless combined view for the pair** (chosen).

## Decision

At most one `ArtifactIndex` instance exists per resolved physical repository root per process: the shared accessor is keyed strictly by resolved root, so there are at most two instances — engagement and enterprise. Each instance is the [Canonical Per-Repo Artifact Index](../../../projects/platform-core/model/application/data-object/DOB@1783266368.4f-C9z.canonical-per-repo-artifact-index.md) for its root: the single consistency boundary owning that root's in-memory store, SQLite index, and locking. The storage engine and concurrency scheme inside each instance are recorded in [@DOC:ADR@1783406825.9bDM6y.read-model-on-a-single-sqlite-backend-with-copy-on-write-concurrency#decision](ADR@1783406825.9bDM6y.read-model-on-a-single-sqlite-backend-with-copy-on-write-concurrency.md).

Any request spanning both repositories is served by a `CombinedArtifactView` that *wraps* the two singletons and never creates a third index. The view is a thin, stateless reader: it delegates identity resolution to whichever instance owns an artifact, concatenates or merges pair-wide results, and composes a pair-level read-model version (summed generations, joined etags) so change detection keeps working unchanged.

The combined accessor takes exactly two required positional roots — deliberately narrower than a list parameter — and those roots must come from workspace resolution, whose return type already encodes the pair constraint. A fitness test guards this discipline, asserting no call site constructs the combined view from an arbitrary, hand-built path pair.

## Consequences

**Positive:**
- Physical content is parsed, held, and indexed exactly once per root — bounded resource cost regardless of how many scope combinations callers request.
- The divergent-stale-index bug class is eliminated structurally: there is no second instance for the same root to fall behind.
- The two-positional-roots signature plus fitness test makes the workspace invariant machine-checkable instead of conventional.
- Combined reads dispatch SQLite-backed queries to both underlying instances concurrently, preserving the [Concurrent Reads, Serialized Writes](../../../projects/motivation-narrative/model/motivation/requirement/REQ@1782080517.IIl8-4.concurrent-reads-serialized-writes.md) requirement rather than serializing an already-concurrent read path.
- Externally observable behavior is unchanged — callers see the same port surface whether backed by one instance or the combined view.

**Negative / risks:**
- The combined view must implement the full store port method-by-method; each method needs a deliberate merge semantics, and a careless sequential fan-out on a database-backed method doubles combined-path latency.
- Short ids can collide across the two repositories even though full ids do not; pair-wide identity lookups must return both candidates rather than assume uniqueness.
- The composite read-model version cannot be decomposed back into per-repo parts; any future caller needing per-repo change detection must query the singletons directly.
- The fitness test enforces call-site discipline textually, not at runtime — a novel call path that evades its pattern could still construct the view from unresolved paths.
