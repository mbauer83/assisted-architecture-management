## Context

Diagrams contain elements that relate to the architectural model in varied ways: a box may directly represent a model entity, summarize several model connections, or merely scope the view. That relation is situative — it varies per diagram and often per element instance — so the question is where the correspondence lives and whether diagram edits may change the model. The main alternatives considered were:

- **Type-encoded correspondence**: each diagram entity type implies fixed model semantics (a C4 container element "is" an application component). Simple, but it cannot express per-instance variation, forces a new diagram type for every correspondence nuance, and entangles notation with model meaning.
- **Implicit model writes from diagram edits**: diagram creation and editing emit model connections as a side effect (as C4 scope handling once did with containment relations). Convenient, but the model mutates without review, deletions and reorganizations of views silently rewrite architectural facts, and no diff exists for the change that matters most.
- **Backward-compatible layering**: keep the old per-element fields and side-effect paths alive alongside a new mechanism. Avoids migration, but perpetuates two sources of truth and lets the forbidden write path survive indefinitely.
- **Explicit per-diagram binding records, with a one-time migration** (chosen): correspondence lives on first-class binding records in the diagram, and the implicit write path is removed entirely.

## Decision

The model, the diagram, and their correspondence form three separate graphs: the model graph M (persistent architectural entities and relationships), the diagram graph D (diagram-owned nodes, edges, layout, and interaction state), and the bindings B — explicit records stored in diagram frontmatter that relate the two. A binding names its subject (a diagram element, or the diagram itself), a correspondence kind (`represents`, `abstracts`, or `scoped-by`), and an id-based model target. Correspondence never lives on the diagram-element type: each [Diagram Type Definition](../../../projects/platform-core/model/business/business-object/BOB@1777390172.7gJz0U.diagram-type-definition.md) declares which model types an element *may* bind to and which correspondence kinds are meaningful, but the chosen correspondence is per instance. Binding integrity — subject resolution, target resolution, admissible kinds, cardinality rules — is checked per diagram type through [Pluggable Diagram-Type Verification & Rendering](../../../projects/motivation-narrative/model/motivation/requirement/REQ@1781976357.A5WgC8.pluggable-diagram-type-verification-rendering.md).

The core invariant: diagram authoring never mutates the model as a side effect. A binding only *states* correspondence; when a diagram interaction should genuinely produce a model relationship, that happens exclusively through an explicit materialization step with a dry-run preview and a separate commit. The former implicit path — C4 create/edit emitting containment connections into the model — is removed outright rather than shimmed, and existing repositories are converted by a one-time migration that synthesizes bindings from legacy per-element fields and reports every previously auto-emitted model connection before removal.

## Consequences

**Positive:**
- The model changes only through reviewed, explicit operations; every model mutation has a visible diff and an intentional author.
- Many-to-many, partial, and abstraction-like correspondences are expressible: one element can abstract several model connections; unbound sketch elements are simply elements without bindings.
- New diagram types add correspondence rules declaratively, without touching generic diagram machinery.
- Verifier rules make integrity first-class: dangling subjects or targets are structural errors, not silent drift. The same explicit-correspondence discipline generalizes to other view families, such as the invariant behind [Datatype–Data Object Relationship Consistency](../../../projects/motivation-narrative/model/motivation/requirement/REQ@1781704601.sbkuwf.datatype-data-object-relationship-consistency.md).

**Negative / risks:**
- Authoring gains a step: users who expect a diagram edit to "just create" model relationships must run the materialization preview and commit.
- The migration is a hard cut — diagrams either use bindings or fail validation; there is no compatibility mode.
- Bindings are additional persisted data that can proliferate; cardinality rules and cascade-on-delete must be enforced to keep them coherent.
- Derived views carry no sync state; keeping a derived diagram current requires an explicit refresh rather than automatic propagation.
