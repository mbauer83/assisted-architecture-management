## Context

The backend's domain layer had accumulated outward dependencies: catalog lookups reached a global module registry in infrastructure, path classification imported configuration code, and ontology data was imported from a concrete ontology package. This coupling made the domain untestable without global state, blocked hosting two configured registries in one process, and undermined the [Extensibility and Configurability](../../../projects/motivation-narrative/model/motivation/principle/PRI@1712870400.uraDPR.extensibility-and-configurability.md) principle that ontologies and diagram types are swappable modules. The main alternatives considered for breaking the dependency were:

- **A global service locator relocated into the domain**: moving the registry accessor into the domain package removes the import arrow but keeps the dependency — hidden coupling, global mutable state, initialization-order fragility, cross-test contamination, and caches that outlive registry replacement. A locator violates dependency inversion regardless of which package the file sits in.
- **Threading a registry parameter through every call site**: functional purity without inversion benefit — dozens of catalog functions and their caches would take a never-changing singleton as a parameter, adding pervasive noise while breaking memoization.
- **Convention-only layering**: document the rules and rely on review. Layering conventions without a failing test demonstrably drift; the existing violations arose under exactly this regime.
- **Focused injectable catalogs wired at composition roots, with a test-enforced dependency matrix** (chosen).

## Decision

The [Architecture Backend](../../../projects/platform-core/model/application/application-component/APP@1777293133.OYEmP1.architecture-backend.md) follows strict hexagonal layering governed by an explicit dependency matrix: the domain imports domain only; the application layer imports domain and application; `config` is a leaf that may use domain value types but nothing else; ontology packages are pure plugins depending only on domain contracts; diagram-type packages are adapter-plugins additionally permitted to touch infrastructure renderers; infrastructure imports inward. Only the composition roots — the FastAPI bootstrap, MCP server entry points, and CLI mains — wire layers together.

Registry semantics are expressed as narrow domain protocols (ontology catalog, connection semantics, diagram-type catalog) with immutable implementations backed by the frozen [Module Catalog](../../../projects/platform-core/model/application/application-component/APP@1712870400.yNhgdh.module-catalog.md). A mutable builder populates the catalog at startup and rejects registration after freeze; the resulting catalogs are injected by constructor or parameter from the composition roots, never resolved from a global. Derived values are cached inside each catalog instance, so replacing a catalog replaces its caches atomically. The matrix is enforced by an AST-based architecture fitness test that covers all packages, including lazy in-function imports; it ships with a baseline recording pre-existing violations, fails on any new violation, and the baseline is burned down to empty.

## Consequences

**Positive:**
- The domain is pure: unit tests construct catalogs from minimal fixtures with zero global state and no cross-test contamination.
- Dependencies are visible in signatures; there is no hidden coupling through a locator, and two differently configured catalogs can coexist in one process.
- The architecture is self-defending: a new layering violation fails CI immediately instead of surfacing in a later audit.
- Plugin boundaries are explicit — adding an ontology or diagram type cannot silently entangle the core, keeping module extensibility credible.
- Frozen catalogs eliminate initialization-order bugs and stale-cache hazards from registry replacement.

**Negative / risks:**
- Composition roots carry more wiring code, and every new capability must be threaded through injection rather than grabbed globally — a real but deliberate friction.
- The AST test is custom machinery that must itself be maintained as import idioms evolve; clever dynamic imports could evade it.
- The baseline mode tolerates old violations until burn-down completes, so the guarantee is only as strong as the discipline to empty it.
- Diagram-type packages retain a sanctioned dependency on infrastructure renderers; the adapter-plugin classification is a pragmatic exception that must not widen.
