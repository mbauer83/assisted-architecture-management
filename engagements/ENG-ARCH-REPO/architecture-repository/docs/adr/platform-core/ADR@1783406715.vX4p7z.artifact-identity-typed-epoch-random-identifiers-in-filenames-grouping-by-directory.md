## Context

Every artifact in the repository — entity, connection, diagram, document — needs an identity that survives renames, merges from concurrent authors, and reorganization of the repository tree. Because artifacts are plain files in git — the storage substrate recorded in [@DOC:ADR@1780761609.GQWvwi.markdown-file-based-architecture-repository#decision](ADR@1780761609.GQWvwi.markdown-file-based-architecture-repository.md) — identity and location interact: a scheme that couples them makes every move a breaking change. Users also need a way to group related artifacts (model-projects, diagram and document collections) as the repository grows. The main alternatives considered were:

- **UUIDs as identifiers**: globally unique without coordination, but opaque — a UUID in a diff, a connection file, or a diagram reference tells a reviewer nothing about what it points at, degrading exactly the human-readable review workflow the file-based repository exists to serve.
- **Sequential identifiers** (e.g., `REQ-042`): compact and readable, but allocation requires a central counter; concurrent agents and branch-based workflows produce duplicate ids and merge conflicts on the counter itself.
- **Frontmatter-based grouping**: a `group:` field inside each file would express membership, but it is invisible in the directory tree, turns every regrouping into a content edit across many files, and inevitably drifts apart from where the file actually lives.
- **Typed epoch-random identifiers in filenames, with directory-based grouping** (chosen): identity is `TYPE@epoch.random.friendly-slug`, baked into the filename; grouping is expressed purely by directory placement.

## Decision

Every artifact's identity is its filename prefix `TYPE@epoch.random` (e.g., `REQ@1712870400.Kk6Ll6`), followed by a human-readable slug that may change freely. The typed prefix is the stable identity: all structural references — connections, diagram bindings, cross-document links — resolve by id, never by path, so an artifact is location-independent and rename-safe. Identifiers are allocated by the [Identifier Allocation Service](../../../projects/platform-core/model/application/application-component/APP@1781976554.AeU7V3.identifier-allocation-service.md) without any central coordinator: the creation epoch plus a random suffix makes collisions between concurrent agents statistically negligible, and uniqueness is checked rather than assumed, satisfying [Verified Unique Identifiers](../../../projects/motivation-narrative/model/motivation/requirement/REQ@1712870400.Kk6Ll6.verified-unique-identifiers-for-entities-connections-diagrams-and-documents.md). The type prefix and slug keep the id meaningful in diffs, logs, and diagram source.

Grouping is expressed by directory placement, never by a frontmatter field. Model-projects wrap the model tree; diagram and document collections sit below their family roots. Re-homing an artifact is a `git mv`; because references are id-based, moves cannot break them, which is what makes [Independent Artifact Grouping](../../../projects/motivation-narrative/model/motivation/requirement/REQ@1780505955.MdtfC3.independent-artifact-grouping.md) safe: the directory is the single source of truth for membership, the indexed group is derived from the path, and there is no stored field to validate or keep in sync.

## Consequences

**Positive:**
- Rename-safe identity: slugs and directories can be reorganized at will; the `TYPE@epoch.random` prefix and every reference to it stay valid.
- No allocation coordination: concurrent human and AI authors on separate branches create artifacts without a counter service or merge conflicts over ids.
- Human-readable everywhere: type prefix plus slug make ids legible in diffs, file listings, and PlantUML source.
- Grouping is visible in the tree and git-native: membership changes are `git mv` operations with clean, reviewable diffs, and subtrees can be inspected with ordinary filesystem tools.
- No group/location drift: membership cannot disagree with placement because placement is the membership.

**Negative / risks:**
- Identifiers are long; raw filenames are bulkier than sequential ids, and tooling must render friendly names on top of them.
- Relative-path links inside documents are the one path-coupled reference form and must be rewritten transactionally when files move.
- Collision resistance is probabilistic, not guaranteed; verification must still detect and reject the rare duplicate.
- Group membership is exclusive per axis: an artifact lives in exactly one directory, so cross-cutting membership must be modeled as views rather than multi-homing.
