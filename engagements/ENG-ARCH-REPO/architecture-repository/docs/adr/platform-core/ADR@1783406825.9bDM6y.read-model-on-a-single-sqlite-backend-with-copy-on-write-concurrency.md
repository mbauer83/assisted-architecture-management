## Context

The markdown files in git are the source of truth, but queries — full-text search, graph traversal, entity context — need an index. That index must update incrementally on every write (agents write continuously), serve 100+ concurrent readers without starving them behind a writer, and stay true to the offline, git-native design. The main alternatives considered were:

- **Migration to DuckDB**: attractive for columnar analytical scans, but its FTS extension does not update incrementally — the full-text index must be rebuilt from scratch on any source change. That disqualifies it for a write path that relies on cheap per-write row-level upserts, and columnar analytics is not this read model's bottleneck.
- **A client-server database** (e.g. PostgreSQL): strong concurrency, but an infrastructure dependency that contradicts the offline, git-native, zero-setup design.
- **Per-query filesystem scans**: no index to maintain, but O(repo) latency on every read — unacceptable at repository scale.
- **A single SQLite database per index, with Copy-on-Write concurrency** (chosen): FTS5 gives incremental row-level full-text maintenance, recursive CTEs cover graph traversal, and reader-starvation is solved at the concurrency layer rather than by swapping engines.

## Decision

The query read model is one SQLite database per index — the [Canonical Per-Repo Artifact Index](../../../projects/platform-core/model/application/data-object/DOB@1783266368.4f-C9z.canonical-per-repo-artifact-index.md) — rebuilt from the markdown source at startup and updated incrementally on each write. The per-root singleton discipline governing these instances is recorded in [@DOC:ADR@1783406811.fm8W_z.one-canonical-artifact-index-per-repository-root-a-combined-view-for-the-pair#decision](ADR@1783406811.fm8W_z.one-canonical-artifact-index-per-repository-root-a-combined-view-for-the-pair.md). FTS5 virtual tables provide full-text search with cheap per-row upserts; recursive CTEs provide graph traversal and connection statistics. The persisted shape is modeled in the "Artifact Persistence Model" datatype diagram (DATATY@1782085920.9Nrbqf).

Concurrency uses a Copy-on-Write scheme: readers capture an immutable read-model snapshot (in-memory store plus a consistent database view, bound together with a generation number) and use it lock-free for the read's lifetime, while a single serialized writer prepares the next snapshot and publishes it in one atomic swap. Readers never wait on writers; they see the previous committed generation. Nested secondary indexes are copied functionally so no published snapshot is ever mutated.

There is no multi-backend factory for the read model. Call sites depend on the store port, and a configuration seam exists, but exactly one backend ships; alternative text-index engines are adopted only if measured load demands it.

## Consequences

**Positive:**
- Readers are lock-free and always see a fully consistent snapshot — no partial state, no torn reads across the memory/database boundary.
- Incremental FTS maintenance keeps write latency low; no full index rebuild on change.
- No infrastructure dependency: the index is an embedded, process-local file, and the repository remains fully usable offline.
- The read model is derived and disposable — rebuilt from markdown at startup — so rollback is "revert and restart" with no data migration ever.
- A single backend avoids the complexity and test surface of a backend matrix nobody uses.

**Negative / risks:**
- Writes serialize through a single lock; write throughput is bounded by design and acceptable only because writes are far rarer than reads.
- Copy-on-Write holds two snapshots during a full rebuild, roughly doubling peak memory at that moment.
- Snapshot immutability is enforced by discipline plus dedicated mutation-safety tests; a missed deep copy on a nested index silently corrupts a reader's snapshot.
- SQLite's read-connection pool is a potential ceiling under extreme concurrency; if measurements ever exceed it, a text-index fallback with incremental segment indexing is the planned escape hatch — a second index to keep consistent.
