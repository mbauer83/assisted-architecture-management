## Context

AI agents interact with the repository through MCP, one of the mandated [Tool Interfaces MCP/CLI/REST](../../../projects/motivation-narrative/model/motivation/requirement/REQ@1712870400.peinbQ.tool-interfaces-mcp-cli-rest.md). The surface must let a deployment grant an agent exactly the capability it needs — read-only exploration, model authoring, or access to confidential assurance content — and each tool must be selected and invoked correctly by harnesses that discover tools dynamically rather than loading one upfront list. The main alternatives considered were:

- **One server with mode-multiplexed mega-tools**: a single tool with a `mode` discriminator fanning out to operations with different safety semantics — parameter soup, structurally valid but semantically invalid argument combinations, and no way to carry an accurate per-tool safety annotation when one tool is simultaneously read-only and destructive.
- **Splitting write servers by sub-domain** (model authoring vs. diagram authoring): tidy taxonomy, but model and diagram writes are co-needed in virtually every authoring session, so the split forces two server registrations per session for no differential grant.
- **Compatibility shims when tools are renamed or decomposed**: doubles the surface, keeps deprecated parameters alive, and lets documentation go stale against a contract nobody should use.
- **Servers split only at capability/permission boundaries, tools decomposed by responsibility, breaking changes migrated fully** (chosen).

## Decision

The MCP surface is four servers along capability and permission boundaries, all exposed through the backend's [MCP Interface](../../../projects/platform-core/model/application/application-interface/AIF@1712870400.wuleEe.mcp-interface.md) and served by the single unified backend recorded in [@DOC:ADR@1783406851.pGCuZn.one-unified-backend-authority-every-write-through-the-same-verified-pipeline#decision](ADR@1783406851.pGCuZn.one-unified-backend-authority-every-write-through-the-same-verified-pipeline.md): `arch-repo-read` (safe for read-only agents and low-trust contexts), the [Architecture MCP Write Interface](../../../projects/platform-core/model/application/application-interface/AIF@1780783713.Ik61W4.architecture-mcp-write-interface.md) served as `arch-repo-write`, and the gated pair `arch-assurance-read` / `arch-assurance-write`, which register only when the confidential assurance store is configured and unlocked — fail-closed, so no assurance content is reachable by an agent in a deployment that has not enabled it, and the read server additionally enforces a max-classification ceiling. The confidential tier behind that gate is recorded in [@DOC:ADR@1783406789.I82vuJ.confidential-assurance-tier-separate-gated-store-with-one-way-persisted-references#decision](../assurance/ADR@1783406789.I82vuJ.confidential-assurance-tier-separate-gated-store-with-one-way-persisted-references.md).

The principled trigger for a separate server is a capability or gating boundary, never sub-domain taxonomy: `arch-repo-write` stays unsplit because model and diagram authoring are co-needed in one session and a split would grant nothing differentially.

Within a server, tools are decomposed by responsibility: each tool carries exactly its required parameters (no conditionally-relevant ones, no representable invalid combinations) and an accurate safety annotation — read-only, write, or destructive. Because modern harnesses discover tools dynamically, per-tool correctness matters more than tool count; a few extra focused tools are cheap, a mislabeled multiplexer is not. Descriptions stay concise but never shed safety- or selection-critical semantics. Breaking changes to the surface migrate fully across code, tests, docs, and the self-model; there are no deprecation shims.

## Consequences

**Positive:**
- Permissions are grantable by configuration alone: which server a client wires in determines what it can do, and read-only agents structurally lack a mutate path.
- Confidential assurance content is unreachable by default; the unlock gate plus classification ceiling makes exposure an explicit, logged deployment choice.
- Invalid parameter combinations are unrepresentable by construction, and accurate read-only/destructive annotations let harnesses apply correct safety policies per tool.
- No shim layer means one contract, current documentation, and a surface agents re-discover cleanly each session.

**Negative / risks:**
- Four servers mean four bridge registrations to configure and keep consistent across client setups.
- Breaking changes without shims require disciplined full migration, and MCP signature changes only take effect after a client session restart — there is no live validation window.
- Gated servers are invisible until the store is unlocked, which can read as "missing" rather than "locked" to an unfamiliar operator; structured unavailable/locked status mitigates but does not remove this.
- Responsibility decomposition yields more tools than a multiplexer would; this is acceptable only while harnesses load tools on demand.
