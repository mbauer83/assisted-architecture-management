## Context

Named attribute profiles let one reusable attribute set (a shared `ai-provenance` block, an
`ownership` block) be bound by many specializations, on both entities and connections. Reuse
introduces two new ways a repository's schema layer can be internally inconsistent, and the
question this decision settles is *how each inconsistency should fail*:

- A specialization **binds a profile name that no registry defines**. The registry is then
  ill-defined as a whole — nothing downstream can trust that a bound name resolves.
- Two contributing fragments **disagree on an attribute's `type`** (base vs. a bound profile,
  or two bound profiles). Only the one `(type, specialization)` pair that composes them is
  ambiguous; every other pair still resolves cleanly.

Treating these alike would be wrong in both directions: failing the whole repository for one
scoped type clash denies access to an otherwise-valid model, while letting an undefined
binding through would let the model resolve differently depending on which module happened to
load. The alternatives considered were a single uniform severity (rejected — it cannot be
right for both blast radii), persisting a quarantine list (rejected — see Decision), and
gating only some transports (rejected — a gap in any one transport defeats the guarantee).

## Decision

Classify every conflict by **blast radius** and fail each accordingly.

- **Class A — structural.** An undefined-profile binding aborts startup on an engagement
  repository (the model cannot be trusted to resolve at all) and warns on the enterprise
  repository. It is surfaced at load time, before any write is attempted.
- **Class B — scoped (quarantine).** A type clash quarantines exactly the one
  `(type, specialization)` pair. Reads continue; the [Model
  Verifier](../../../projects/platform-core/model/application/application-component/APP@1712870400.ca3vm7.model-verifier.md)
  reports the clash as a blocking error (`E043` for entities, `E045` for connections), and a
  create or edit onto that pair is refused.

Quarantine is enforced at **a single write boundary** inside the [Architecture
Backend](../../../projects/platform-core/model/application/application-component/APP@1777293133.OYEmP1.architecture-backend.md)
— the one choke point every transport (GUI, REST, MCP) already funnels through under [One
Unified Backend
Authority](ADR@1783406851.pGCuZn.one-unified-backend-authority-every-write-through-the-same-verified-pipeline.md).
A gate at that one point covers all transports at once; per-transport checks would leave a
gap the moment a new transport is added.

Quarantine is **computed on demand and never persisted.** It is a pure function of the
current registry, bindings, and schemata, so it self-clears the instant the conflict is
reconciled — there is no stored quarantine list to drift out of step with the files, and no
migration to run when a conflict is fixed. Reconciliation (`arch-repair`) reports each
quarantining conflict with concrete proposed resolutions (rename, align type, unbind) but
never rewrites operator-authored content automatically.

## Consequences

**Positive:**
- A scoped type clash costs access only to the affected pair; the rest of the model stays
  readable and writable.
- An undefined binding can never resolve silently — it stops the engagement repository at
  startup, where it is cheapest to diagnose.
- One write-boundary gate makes the guarantee transport-independent: a GUI unaware of
  quarantine still cannot write ambiguous data, which is the property the acceptance test
  pins.
- No persisted quarantine state means no reconcile step and no stale-list failure mode; fixing
  the profiles is the whole fix.

**Negative / risks:**
- Recomputing quarantine per write is slightly more work than reading a cached flag; it is
  cheap because resolution is already required on the write path, but it is not free.
- The engagement-hard / enterprise-soft split for Class A is a deliberate asymmetry an
  operator must understand: the same broken binding aborts one tier and only warns on the
  other.
- Class B depends on every write genuinely passing through the single boundary; a future
  write path that bypasses it would silently reopen the gap the single-gate decision closes.
