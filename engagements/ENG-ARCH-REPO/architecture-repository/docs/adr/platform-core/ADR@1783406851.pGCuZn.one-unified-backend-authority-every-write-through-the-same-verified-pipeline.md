## Context

The repository is edited through four surfaces — GUI, REST API, CLI, and MCP servers — by humans and AI agents simultaneously. Every surface must see the same index, and every mutation must be verified against the ontology before it lands; the model's own requirements demand [Write Access Only via Tools](../../../projects/motivation-narrative/model/motivation/requirement/REQ@1712870400.m117_R.write-access-only-via-tools.md) and mandatory [Verification Tools](../../../projects/motivation-narrative/model/motivation/requirement/REQ@1712870400.JTRw1x.verification-tools.md) in the write path. The main alternatives considered were:

- **Per-surface embedded engines**: each surface links the engine and runs it in-process — N processes with divergent in-memory indexes over the same files, racing each other on writes and each caching a different view of the repository.
- **Library-only design**: ship the engine as a library and let every client embed it — the same divergence, and no place to serialize concurrent human and agent writes.
- **Verification as a post-hoc lint step**: write freely, verify later — permits windows where the written state and the verified state drift apart, exactly the inconsistency verification exists to prevent.
- **One long-running backend process as the single runtime authority** (chosen): all surfaces are thin clients of one composition root.

## Decision

Exactly one long-running process — the [Architecture Backend](../../../projects/platform-core/model/application/application-component/APP@1777293133.OYEmP1.architecture-backend.md) — owns every operation that reads or mutates artifacts. The GUI, REST API, CLI, and MCP servers are thin surfaces over it: the MCP stdio entry points are shims that auto-start the backend and forward requests over HTTP to the mounted MCP endpoints, where the [Architecture MCP Endpoint Adapter](../../../projects/platform-core/model/application/application-component/APP@1712870400.kRZYOA.architecture-mcp-endpoint-adapter.md) serves them from inside the backend process (the server topology itself is recorded in [@DOC:ADR@1783406863.eEC2t0.mcp-surface-topology-split-read-write-servers-gated-assurance-servers-responsibility-decomposed-tools#decision](ADR@1783406863.eEC2t0.mcp-surface-topology-split-read-write-servers-gated-assurance-servers-responsibility-decomposed-tools.md)). Artifact-affecting CLI commands call the same REST surface the GUI uses; only bootstrap and configuration commands, which must run before a backend exists, remain independent.

The consequence of single authority is a single pipeline: every write, regardless of surface, flows through the same [Operation Registry](../../../projects/platform-core/model/application/application-component/APP@1777399926.PDJI8F.operation-registry.md), the same serialized write queue, and the same verifier. There is no unverified write path — direct editing of model files is out of contract, and the same operation issued via REST, MCP, or CLI produces the same result and the same verification.

## Consequences

**Positive:**
- One composition root: one in-process cache and index, one write queue serializing human and agent writes, one permission and gating model, one place to reason about confidentiality.
- Verification is structurally unavoidable — no surface can produce an unverified mutation, so the repository never drifts from its verified state.
- Cross-surface parity is testable: the same operation via any surface must yield identical results.
- Thin surfaces stay thin; new surfaces (or new tools on existing ones) inherit the full pipeline for free.

**Negative / risks:**
- The backend is a single point of failure: artifact-affecting CLI commands require it to be running and fail with a clear error when it is absent.
- Code changes require a backend restart to take effect, since all surfaces run against the long-lived process.
- Every operation pays a local HTTP hop; latency budgets must account for it.
- All concurrency concentrates in one process, making its internal concurrency model — recorded in [@DOC:ADR@1783406825.9bDM6y.read-model-on-a-single-sqlite-backend-with-copy-on-write-concurrency#decision](ADR@1783406825.9bDM6y.read-model-on-a-single-sqlite-backend-with-copy-on-write-concurrency.md) — load-bearing for the whole system.
