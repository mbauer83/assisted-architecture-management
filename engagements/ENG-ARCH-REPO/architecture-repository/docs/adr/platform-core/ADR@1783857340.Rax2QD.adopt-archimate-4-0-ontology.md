## Context

[@DOC:ADR@1780761591._mseZr.adopt-archimate-next-ontology#decision](ADR@1780761591._mseZr.adopt-archimate-next-ontology.md)
adopted "ArchiMate NEXT" — a pre-standardization preview of an ontology evolution (domains
instead of layers, behavioral elements and roles unified in a shared `common/` domain, the
physical layer merged into technology) — because no ratified standard existed yet to adopt
directly.

ArchiMate 4.0 has since been ratified as the real OMG/Open Group standard. Its structural
mechanics confirm the speculative choices the "NEXT" ADR made (domains, the `common/`
placement of behavioral elements), and it additionally standardizes several mechanisms this
repository did not yet have: a first-class viewpoint mechanism with enforcement modes, formal
element/relationship specialization, and a model-exchange interoperability format. Continuing
to call the ontology "NEXT" once a ratified standard exists is confusing and inaccurate — the
name refers to a real, dated specification now, not a moving target.

## Decision

Adopt ArchiMate 4.0 (the ratified standard) as the system's ontology, superseding the
"ArchiMate NEXT" naming above. The three-token rename: the ontology package is `archimate_4`,
the module identifier is `archimate-4-0`, and the external alias is `archimate-4` — no legacy
alias is retained, and the prior token inconsistency between the package and the
`permitted_mappings` table is fixed as part of the same change. Self-model content authored
under the "NEXT" name (entity names/slugs, prose descriptions) is swept and renamed to match;
this repository's own model practices what it documents.

The ontology remains a pluggable module behind the frozen module catalog, per
[@DOC:ADR@1783406752.kQ__1X.hexagonal-core-with-a-test-enforced-dependency-policy-no-service-locator#decision](ADR@1783406752.kQ__1X.hexagonal-core-with-a-test-enforced-dependency-policy-no-service-locator.md) —
adopting the ratified standard changes what the module contains, not how it is loaded.

## Consequences

**Positive:**
- Standards-based ontology backed by a ratified specification, giving external legitimacy and
  a real conformance target instead of a moving preview.
- The "NEXT" ADR's speculative structural bets (domains, shared `common/` behavioral placement)
  are validated by the ratified standard rather than superseded by it.
- New standard mechanisms (viewpoints, specialization, model-exchange) become available to
  adopt incrementally, each behind its own decision rather than bundled into this one.

**Negative / risks:**
- The ratified specification text itself is copyrighted (The Open Group) and is never vendored
  into this repository — guidance/spec prose is imported per-repo into gitignored caches via a
  dedicated CLI import command, with enterprise/engagement precedence and provenance tracking,
  never committed in-repo.
- Content authored under the "NEXT" name required a one-time rename sweep (package/module/alias
  tokens, self-model entity names/slugs and prose) to complete the adoption cleanly.
- Third-party ArchiMate 3.x tooling (Archi, BiZZdesign) still does not natively understand this
  repository's `common/` domain convention; interoperability is addressed separately through the
  model-exchange mapping work, not by this decision.
