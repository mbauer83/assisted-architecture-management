## Context

Multiple teams and agents produce architecture content concurrently: exploratory drafts, project-specific detail, and organisation-wide baselines all need a home, but they have different audiences, confidentiality expectations, and quality bars. The repository topology must give each engagement a fast, autonomous working space while maintaining a single curated baseline that other engagements can trust and reuse — satisfying the [Two-Tiered Repository](../../../projects/motivation-narrative/model/motivation/requirement/REQ@1712870400.kOU3al.two-tiered-repository.md) requirement. The main alternatives considered were:

- **One shared repository for everything**: simplest topology, but it offers no confidentiality boundary between engagements, no space for drafts that are not ready for organisation-wide visibility, and every write contends for the same review pipeline — a coordination bottleneck that worsens as agentic authoring volume grows.
- **Fully independent repositories with copy-paste reuse**: maximum autonomy, but copied content drifts immediately, there is no canonical baseline to verify against, and cross-engagement reuse degrades into divergent forks of the same entities.
- **Branch-based staging within one repository**: git branches provide isolation mechanics, but they do not model organisational visibility boundaries — anyone with repo access sees every branch, and long-lived engagement branches accumulate merge debt rather than curated content.
- **Two tiers with an explicit promotion step** (chosen): per-engagement repositories for day-to-day work, one shared enterprise repository that accumulates only promoted content.

## Decision

Architecture work happens in per-engagement repositories; a single shared [Enterprise Repository](../../../projects/promotion-and-tiering/model/business/business-object/BOB@1712870400.6Uok0b.enterprise-repository.md) accumulates only promoted, curated content — one enterprise repository, many engagements. Each [Engagement Repository](../../../projects/promotion-and-tiering/model/business/business-object/BOB@1712870400.so7gfN.engagement-repository.md) is the autonomous draft and detail space for one project context.

Promotion is the explicit governance and confidentiality gate between the tiers, executed by the [Repository Promotion Service](../../../projects/promotion-and-tiering/model/common/service/SRV@1712870400.Uv9Wx9.repository-promotion-service.md) and reachable through the GUI, MCP, and REST. It is a one-way transfer of an explicitly selected set of entities *with their connections* — promoted content is never orphaned from its relationships. Cross-repo naming conflicts are detected by dual matching — same `(artifact_type, normalized_name)` or same `(artifact_type, id_suffix)` — so both a renamed entity and a same-named different entity are caught, with accept-engagement, accept-enterprise, or manual-merge resolution. Engagement groups are explicitly mapped into a flat enterprise namespace as part of the reviewed promotion plan, avoiding cross-engagement collisions.

The verifier enforces the asymmetric-reference rule that makes the tiers sound: enterprise entities may reference only enterprise entities, while engagement entities may reference both. Enterprise changes land through a branch-based review workflow (accumulate → save → submit branch → auto-detect merge), which works on any git host without API integration.

## Consequences

**Positive:**
- Engagements work autonomously at full speed; the enterprise baseline stays deliberately curated rather than accumulating drafts.
- Confidential or immature engagement content never reaches the shared tier implicitly — promotion is the only path up, and it is explicit, selected, and reviewed.
- Cross-engagement reuse has a canonical source: proven content is promoted once and referenced, not copied and forked.
- Conflict detection at the gate catches naming and identity collisions before they corrupt the shared baseline.
- The asymmetric-reference rule keeps the enterprise repository self-contained and independently consumable.

**Negative / risks:**
- Promotion is a manual, deliberate step; content that should be shared can lag in an engagement until someone promotes it.
- The same concept can temporarily exist in several engagements before promotion reconciles it; the dual-match conflict detection mitigates but cannot prevent pre-promotion divergence.
- Group mapping and conflict resolution add review effort to each promotion, proportional to the amount promoted.
- The enterprise branch-review lifecycle adds states (accumulating, pending, synced) that users must understand when enterprise writes appear "not yet on main".
