## Configurable Frontmatter Schemata

The system supports configurable frontmatter schemata specific to each of the three file types: entity files, connection files (.outgoing.md), and diagram files. Each schema defines which frontmatter fields are required, optional, and valid. Fields required by core tooling and processes are mandatory and cannot be removed or overridden — the schema can only extend (add custom fields).

### Implementation

- **Format**: JSON Schema (Draft 2020-12), language-agnostic standard
- **Location**: `.arch-repo/schemata/` within the architecture repository root
- **File convention**: `frontmatter.{file-type}.schema.json` where file-type is `entity`, `outgoing`, or `diagram`
- **Extend-only semantics**: Tool-required base fields are always enforced regardless of schema content; custom schemas can add constraints but cannot remove required fields
- **Default**: Free schema — when no schema file exists for a file type, no additional validation beyond tool-required fields
- **Validation library**: Python `jsonschema` (Draft202012Validator)
- **Integration**: `model_verifier_rules.check_frontmatter_schema()` validates frontmatter against the schema; issues are reported as warnings (W041) for incremental adoption
- **Write scaffolding**: `model_write_formatting.format_entity_markdown()` accepts `repo_root` to scaffold from schema
