## Configurable Model Attribute Schemata

The system supports configurable attribute schemata for file content sections, defined per entity-type and per connection-type. Attribute schemata govern the valid attributes in Properties tables (for entities) and in connection metadata blocks (for connections in .outgoing.md files). The default is a free schema — no required attributes beyond the core frontmatter.

### Implementation

- **Format**: JSON Schema (Draft 2020-12), language-agnostic standard
- **Location**: `.arch-repo/schemata/` within the architecture repository root
- **File convention**: `attributes.{artifact-type}.schema.json` (e.g., `attributes.driver.schema.json`, `attributes.requirement.schema.json`) for entity types; `connection-metadata.{connection-type}.schema.json` for connection types
- **Default**: Free schema — when no schema file exists for an entity-type or connection-type, Properties table attributes are unconstrained
- **Schema capabilities**: Required attributes, optional attributes, allowed value enums, string patterns, additional properties allowed by default
- **Validation library**: Python `jsonschema` (Draft202012Validator)
- **Integration**: `model_verifier_rules.check_attribute_schema()` extracts Properties table key-value pairs and validates against the per-type schema; issues reported as warnings (W042) for incremental adoption
- **Write scaffolding**: When a schema exists for the entity type, `model_write_formatting.format_entity_markdown()` scaffolds the Properties table with required and optional attribute rows

### Distinction from Frontmatter Schemata

Frontmatter schemata are per file-type (entity, outgoing, diagram) and govern YAML frontmatter fields. Attribute schemata are per entity-type or connection-type and govern content-section attributes (Properties tables and connection metadata blocks).
