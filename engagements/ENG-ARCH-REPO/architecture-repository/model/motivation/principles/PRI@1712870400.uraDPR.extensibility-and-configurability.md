## Extensibility and Configurability

The architecture repository system shall be extensible and configurable at multiple levels. Repository behavior — including frontmatter schemata (per file-type: entity, connection-file, diagram — extend-only), attribute schemata for content sections (per entity-type and per connection-type, defaulting to free schema), valid entity types, connection type rules, and directory structure conventions — shall be configurable via git-based configuration files at both the enterprise level and the engagement/project level. The ontology shall be extensible beyond the base ArchiMate NEXT element types and connection types. This principle ensures the system can adapt to diverse organizational contexts without forking the core tooling.

## Properties

| Attribute | Value |
|---|---|
| Priority | Must |
| Rationale | Different organizations and engagements have different metadata needs, ontology extensions, and governance requirements |

## Notes

Configuration flows from enterprise to engagement: enterprise configuration provides defaults and constraints; engagement configuration may extend or specialize within those bounds.
