## Canonical Per-Repo Artifact Index

Represents the one-instance-per-physical-root caching invariant inside the Architecture Backend's index layer. Before this invariant was enforced, a request for {engagement}, {enterprise}, or {engagement, enterprise} combined could each allocate a fully-separate, fully-populated index instance covering the same physical content — up to three redundant copies of engagement-repo data in the worst case. Now the index layer's shared-instance cache is keyed strictly by single physical root, so at most two such instances ever exist; a combined-scope read is served by Compose Combined-Scope Read, a stateless composition over the two canonical instances rather than a third instance. Each canonical instance is its own aggregate root and consistency boundary — it owns one SQLite Index for search acceleration, and a write only ever updates the one instance owning the changed path.
