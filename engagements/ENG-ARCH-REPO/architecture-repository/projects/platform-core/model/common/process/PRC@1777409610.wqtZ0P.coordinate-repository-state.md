## Coordinate Repository State

System-level process that serializes write requests through a queue, ensures every committed change (individual write or bulk commit) triggers an index refresh, and notifies connected clients of updated model state. The read/write index can exist as multiple independently-cached instances for the same physical repository — one per distinct root-set scope (e.g. engagement-only vs. engagement+enterprise-combined) — so "every committed change triggers an index refresh" is only true if a write's index update is broadcast to every such instance, not just the one the writer happened to resolve. This was violated for months: several write paths called the index's own update method directly, silently leaving sibling-scoped instances stale until a manual admin reindex (e.g. a GUI-created entity invisible to an MCP delete tool). Fixed by routing every write-commit path through the index layer's own change-broadcast primitive, which already updates every live cached instance sharing the changed path — that primitive existed but had only ever been wired up for externally-driven (git-sync) changes, never the application's own writes. An automated architectural-fitness test now guards against a future write path bypassing the broadcast again.
