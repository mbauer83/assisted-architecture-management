## Coordinate Repository State

System-level process that serializes write requests through a queue, ensures every committed change (individual write or bulk commit) triggers an index refresh, and notifies connected clients of updated model state. The read/write index now exists as exactly one canonical instance per physical repository root — at most two per running backend (engagement, enterprise) — never one per requested scope combination; a combined engagement+enterprise read is served by Compose Combined-Scope Read, a stateless composition over the two canonical instances rather than a third, separately-cached one. A write only ever updates the one canonical instance owning the changed path; the change-broadcast primitive that keeps every live cached instance in sync (fixed after being wired up only for git-sync-driven changes, never the application's own writes) remains in place as defense-in-depth for whatever narrower-scoped instances still exist outside this canonical pair (e.g. short-lived staging indexes) — it is no longer compensating for redundant full-repo duplication. An automated architectural-fitness test guards against a write path bypassing the broadcast, and a second fitness test guards against the per-root keying being widened back to accept multiple physical roots.
